Deploy & embed

appctl serve is a standalone HTTP + WebSocket daemon. Any client that can send an HTTP request or open a WebSocket can talk to it.

The API contract

These are the endpoints. Field names are exact.

EndpointRequestResponse
POST /run {"message":"...", "read_only":true} JSON with answer and events array
WS /chat {"message":"..."} or plain string One JSON line per AgentEvent
GET /tools none JSON array of registered tools with JSON Schema
GET /schema none Full .appctl/schema.json as JSON
GET /history none Recent executions from the local SQLite audit DB
GET /config/public none Non-secret server config (flags, mode, provider name)

Pass auth via Authorization: Bearer <token>, header x-appctl-token, or ?token= on WebSocket, when serve is started with --token.

Per-request safety overrides are accepted on POST /run:

POST /run
{
  "message": "delete all draft orders",
  "dry_run": true,
  "read_only": false,
  "confirm": false
}

These override the server defaults for that one request only.

Run locally

# sync first
appctl sync --openapi https://api.example.com/openapi.json

# start the daemon
appctl serve --port 4242

# bundled web UI
open http://127.0.0.1:4242/

Internal LAN / Tailscale

appctl serve \
  --bind 0.0.0.0 \
  --port 4242 \
  --token $(openssl rand -hex 16)

Team members open http://appctl.internal:4242/ for the bundled web UI, or point the VS Code extension at the same address.

Put TLS termination at a reverse proxy (Caddy, nginx) before exposing beyond a trusted network.

Embed in your product

Run appctl serve on a trusted host reachable by your frontend. Your web or mobile app sends prompts and streams events back. Here is a minimal fetch example: 

const res = await fetch("http://appctl.internal:4242/run", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-appctl-token": token,
  },
  body: JSON.stringify({ message: "show me the 5 latest orders" }),
});
const { answer, events } = await res.json();

For streaming, open a WebSocket to /chat:

const ws = new WebSocket("ws://appctl.internal:4242/chat?token=" + token);
ws.send(JSON.stringify({ message: "show me the 5 latest orders" }));
ws.onmessage = (e) => {
  const event = JSON.parse(e.data);
  // event.type: "ToolCall", "ToolResult", "Text", "Done", "Error"
};

The VS Code extension under extensions/vscode is a complete worked example of this WebSocket client pattern.

Safety defaults

appctl serve exposes flags that apply to every request unless overridden:

--read-onlyBlocks all mutating tools at the server level
--dry-runPlans actions but makes no real API calls
--confirmAuto-approves mutations without prompting (default on for serve)
--strictRefuses Inferred HTTP tools until appctl doctor --write verifies them

Threat model

  • Token auth. Set --token for any network beyond localhost. Without it, any client on the network can send prompts.
  • Upstream RBAC. The agent uses your credentials. It cannot bypass your API's authorization. If the key cannot do it, the agent cannot do it.
  • Read-only / dry-run. Server-level defaults reduce blast radius for shared deployments.
  • Audit log. Every tool execution is written to a local SQLite database. Replay with appctl history.
  • Strict mode. Prevents the agent from calling routes that have not been verified by appctl doctor --write.
  • Secrets. API keys live in the OS keychain; nothing secret is written to disk by default.
