Deploy & embed

The CLI is for integrators. appctl serve is how clients (web UI, VS Code, Slack, mobile) talk to the same agent core.

Internal LAN / Tailscale

Run appctl serve --bind 0.0.0.0 --port 4242 --token $(openssl rand -hex 16) on a VM or container reachable on your LAN or Tailscale network. Team members open http://appctl.internal:4242/ for the bundled web UI, or point the VS Code extension at the same URL.

Prefer TLS termination at a reverse proxy (Caddy, nginx) when exposing beyond a trusted network.

Embed in your product

Your app talks to the same HTTP + WebSocket API as the bundled UI:

  • POST /run — JSON { "message": "...", "read_only": true }; response includes events (agent trace).
  • WS /chat — send JSON {"message":"...","dry_run":true} or a plain string; each event is one JSON line (see AgentEvent in the Rust crate).
  • GET /tools, GET /history, GET /schema, GET /config/public — for dashboards.

Pass auth via Authorization: Bearer <token>, header x-appctl-token, or ?token= on WebSocket when serve was started with --token.

Hosted option (future)

A managed, multi-tenant hosted control plane is not part of this repository yet. Today, you self-host appctl serve next to the systems you control.

Threat model (short)

  • Token auth. Set --token for any network beyond localhost.
  • Upstream RBAC. The agent uses your credentials; it cannot bypass your API’s authorization.
  • Read-only / dry-run. Server defaults + per-request overrides reduce blast radius.
  • Audit. Tool executions are logged to local SQLite; replay with appctl history.
  • Strict mode. Refuses HTTP tools that are still Inferred until appctl doctor --write verifies routes.
