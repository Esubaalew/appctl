One command. Any app. Full AI control.

appctl introspects any web app, database, or service and turns it into a sandboxed tool layer that any LLM — and any human typing plain English — can safely drive. The CLI is for setup. Daily use is a conversation. 

cargo install appctl

The 30-second mental model

  1. Sync once. Point appctl at an OpenAPI spec, a Rails / Laravel / ASP.NET / Strapi / Supabase / Django project, a Postgres / MySQL database, or a live URL with a login form. It extracts every resource and action into a normalized schema.
  2. Get a tool layer for free. The schema becomes a set of JSON-schema tools that any LLM can call — OpenAI, Anthropic, OpenRouter, NVIDIA NIM, Groq, Ollama, vLLM, anything.
  3. Humans talk, agent executes. End users type English. The LLM picks the right tool, the executor makes the real HTTPS / SQL call with your real credentials, and every action is logged. Nobody memorises CRUD commands.

Think of it as: OpenAPI / your framework → tools → chat. The CLI is only used by the person doing the one-time integration.

An end-to-end example

Acme runs a Rails storefront in production at https://api.acme.com. Support wants to handle refunds in chat instead of clicking through the admin panel.

1. The dev wires it up (one time)

$ appctl sync --openapi https://api.acme.com/openapi.json
Synced: 42 resources, 187 tools.

$ appctl auth login acme \
    --client-id $CLIENT_ID \
    --auth-url https://acme.com/oauth/authorize \
    --token-url https://acme.com/oauth/token \
    --scopes "orders:write customers:read"
Tokens stored in system keychain.

$ appctl serve --port 4242
Listening on 127.0.0.1:4242  (HTTP + WebSocket + web UI)

2. A support agent talks to it — no CLI, no CRUD

From the VS Code chat panel, a Slack bot, or an internal web UI — anything that speaks to appctl serve:

Find the order from last Tuesday for jane@acme.com and refund $20.
→ list_customers {"email":"jane@acme.com"}
← 1 match, id=7421
→ list_orders {"customer_id":7421,"created_after":"2026-04-14","created_before":"2026-04-15"}
← 1 match, id=98812, total=$129.00
→ create_refund {"order_id":98812,"amount_cents":2000} mutation — confirm?
← refund_id=rfd_88af
Issued a $20 refund on order #98812. Jane will see it in 3–5 business days.

The support agent never saw JSON. The tool trace above is optional — you can hide it, show it, or stream it for debugging.

Four ways people actually talk to it

Setup is CLI. After that, pick whichever surface fits the user:

Terminal REPL

For devs, ops, and anyone who already lives in a shell.

$ appctl chat
you> list the last 5 orders over $500 and export them as CSV
agent> ...

VS Code extension

Chat panel inside the editor with live tool-call traces. Connects to appctl serve over WebSocket. Ships with the repo under extensions/vscode.

HTTP + WebSocket daemon — embed it in anything

Run appctl serve on a trusted box. Your frontend — a Slack bot, an internal admin page, a mobile app, LangChain, Home Assistant, whatever — sends prompts and streams back tool events + the final answer. 

# one-shot
POST /run            { "prompt": "Create a coupon SUMMER 15% off, expires Aug 31" }
GET  /tools          list the registered tools (JSON schema)
GET  /history        recent executions
# streaming
WS   /chat           prompt in, tool events + final answer out

That's the full public surface. Any chat UI you already have can wire into it in a few hours.

One-shot / cron

appctl run "Archive every order older than 180 days"
# exits 0 on success, prints the tool trace + answer

Sources it understands out of the box

  • --openapi OpenAPI / Swagger
  • --rails Ruby on Rails
  • --laravel Laravel
  • --aspnet ASP.NET Core
  • --strapi Strapi
  • --supabase Supabase / PostgREST
  • --django Django
  • --db PostgreSQL / MySQL
  • --url Live URL + login / CSRF
  • --mcp MCP servers
  • --plugin Dynamic plugins

Anything not on this list? Write a 100-line dynamic plugin.

LLM providers

Provider-agnostic. Anything with an OpenAI-compatible API plus native Anthropic: OpenAI, OpenRouter, NVIDIA NIM, Groq, Together, Fireworks, Ollama, LM Studio, vLLM, LiteLLM, Anthropic. Swap the provider in .appctl/config.toml; no code changes. Secrets live in the OS keychain, with env-var fallback for CI/headless.

Production & safety

This is built for real production apps, not toy localhost. Your upstream app's own auth and RBAC still apply — appctl uses your real credentials via API key, Basic, Bearer, or OAuth2 PKCE with refresh. On top of that:

  • --read-only blocks every write tool.
  • --dry-run shows what would happen, no side effects.
  • Mutations prompt by default; --confirm auto-approves.
  • OAuth tokens live in the OS keychain, never on disk.
  • Every tool call logged to a local SQLite audit DB.
  • appctl history replays what happened and when.

Plugins

Anything appctl doesn't cover natively you add as a Rust cdylib against appctl-plugin-sdk: 

appctl plugin install <path|crate|git-url>
appctl plugin list
appctl sync --plugin airtable

Reference implementation: examples/plugins/appctl-airtable.

Will my app work?

appctl only exposes tools it can justify. Framework sync checks for a real API surface (for example Django REST Framework). When in doubt, run appctl doctor after sync to probe routes, or use --strict on chat / run / serve to block unverified HTTP tools until you verify them.

Source Needs If it won’t sync
--openapi Valid OpenAPI 2/3 (URL or file) Fix the spec or point at /openapi.json.
--django Django REST Framework in INSTALLED_APPS Plain Django templates/session apps: use --url for browser login flows, or add DRF and re-run sync.
--rails routes.rb with API routes (or similar) we can parse If no API routes are found, sync warns and skips phantom HTTP tools; use --openapi or --url.
--laravel routes/api.php (or parsed API routes) No routes parsed → warnings only; prefer --openapi if you publish a spec.
--aspnet Swagger/OpenAPI artifact or [ApiController] Publish swagger.json or fall back to --openapi / --url.
--strapi / --supabase Project layout + REST/OpenAPI we can reach Check base URL and anon keys; see Strapi / Supabase docs.
--db Postgres or MySQL connection string SQL tools are introspected from the live schema.
--url Login page + forms we can automate Use for session/HTML apps when there is no machine-readable API.
--mcp MCP server URL See MCP limits.

FAQ

Will my Django / Rails app get fake /api tools?

Not silently. Django sync requires DRF in settings; otherwise HTTP tools are omitted and a warning is stored in schema metadata. Rails/Laravel/ASP.NET sync warns when no API routes are detected. Run appctl doctor --write to probe and mark routes as verified where possible.

Do end users need to know any CLI commands?

No. The CLI is for the person doing the one-time integration (sync, auth login, serve). After that, end users type English into appctl chat, the VS Code panel, or any UI you point at appctl serve. They never say POST /customers — they say "add a customer named Jane".

Does this work against production or only local?

Production. Give it an OpenAPI URL, a Supabase URL, a production PostgreSQL connection string, or a live domain with a login form. Auth is real: API keys, Basic, Bearer, OAuth2 PKCE with refresh. Running it against a localhost toy is just a special case.

How do I embed this in my own product or SaaS?

Run appctl serve next to your backend (a server, a container, a VPC jumpbox). Your frontend — web, mobile, Slack bot, whatever — sends prompts over HTTP (POST /run) or WebSocket (/chat), streams back tool events, and renders them however you like. The VS Code extension is a worked example; copy its WebSocket client into your own app and you're done. See also Deploy & embed for LAN/Tailscale and a short threat model.

What stops the LLM from doing something destructive?

Three layers:

  1. Your upstream app's existing auth and RBAC still apply — appctl uses your credentials, so anything the key can't do, the agent can't do.
  2. appctl flags: --read-only, --dry-run, mutation prompts.
  3. Every call hits a local SQLite audit DB you can replay with appctl history.
Where are my secrets stored?

The OS keychain: macOS Keychain, Windows Credential Manager, Secret Service / gnome-keyring on Linux. Env-var fallback is supported for CI and headless boxes. Nothing secret hits disk by default.

Which LLM should I pick?

Whichever you want — it's provider-agnostic. For private data, point it at a local model via Ollama or vLLM. For quality, OpenAI / Anthropic / NVIDIA NIM all work. Swap by editing one line in .appctl/config.toml.

How does it handle apps without an OpenAPI spec?

Several fallbacks. --rails reads schema.rb + routes.rb. --laravel reads migrations + api.php. --aspnet reads swagger.json or scans [ApiController] attributes. --db reads straight from the database. --url logs in to a live site, walks forms, and extracts CSRF tokens.
