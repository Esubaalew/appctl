Talk to your app. In plain English.

You already have an app. Maybe a Django site. Maybe a Postgres database. Maybe a Laravel API. appctl puts a chat box on top of it so you can say things like "show me the last five orders over $500" and get a real answer from your real data, not a made up one. 

cargo install appctl

What is appctl?

A command line tool that you run once to connect your app to a language model. After that, anyone on your team can open a chat window and ask questions or trigger actions. The chat window talks to the model. The model decides which of your real API endpoints (or SQL queries) to call. appctl actually calls them, with your real credentials, and shows you what happened.

You do not need to expose every button of your app. The model only sees the tools appctl discovered from your code. If your app never had a "delete user" endpoint, the chat cannot invent one.

Three things to picture:

  • Your app stays your app. No rewrite, no SaaS in between. appctl runs on your laptop or your own server.
  • Any LLM works. OpenAI, Anthropic, a local model through Ollama, NVIDIA NIM, Groq. Change one line in a config file to switch.
  • Every call is logged. There is a local SQLite file that records every tool the AI ran. You can replay it later.

How it works in three steps

  1. 1. Sync. Point appctl at your app. It reads your OpenAPI spec, or your Django models, or your Postgres tables, and writes a file that lists every action your app can do.
  2. 2. Serve (or chat). Run appctl chat in a terminal, or appctl serve to get an HTTP endpoint and a built in web chat at http://localhost:4242.
  3. 3. Ask. Type what you want in English. The AI picks the right action. appctl runs it, shows the result, and writes a row to the audit log.

Install

The easiest way, if you have Rust:

cargo install appctl

From source:

git clone https://github.com/esubaalew/appctl
cd appctl
cargo install --path crates/appctl --locked

Check it:

$ appctl --version
appctl 0.2.0

Try it in five minutes

The repo ships with demo apps you can clone and run. Pick one that matches your stack. Every command below is copy and paste real. The output shown is the real output from this machine.

FastAPI (Python): the fastest path

You get a tiny Python server with one endpoint. appctl reads its OpenAPI document and gives the chat one tool. 

git clone https://github.com/esubaalew/appctl
cd appctl/examples/demos/openapi-fastapi

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --host 127.0.0.1 --port 8000 &

appctl sync --openapi http://127.0.0.1:8000/openapi.json \
  --base-url http://127.0.0.1:8000 --force

Real output:

Synced Openapi: 1 resources, 1 tools written to .appctl

Full instructions including appctl chat usage are on the OpenAPI page.

Django REST Framework

Works out of the box with a Django project that has rest_framework in INSTALLED_APPS. appctl reads your models and generates five tools per model (list, get, create, update, delete). 

cd appctl/examples/demos/django-drf
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver 127.0.0.1:8001 &

appctl sync --django . --base-url http://127.0.0.1:8001/api --force

Real output:

Synced Django: 2 resources, 10 tools written to .appctl

Two things matter in the --base-url: you have to include the /api prefix (the sync does not guess it), and the demo has rest_framework.permissions.AllowAny, so you do not need a token. Real Django apps do.

Postgres or MySQL directly

No API needed. appctl connects to the database, reads the tables, and gives the chat SQL tools. 

appctl sync --db "postgres://user:pass@host:5432/dbname" --force
appctl chat "how many orders do we have this month"

Use a read only database user for this. Or run appctl chat --read-only, which blocks every mutating tool at the appctl level.

What stops the AI from doing something bad

Four things:

  • Your app's own permissions. appctl uses the API key or login you gave it. If that key cannot delete users, neither can the agent.
  • Flags you control. --read-only blocks writes. --dry-run plans actions without calling. --strict refuses any unverified route.
  • Prompting on mutations. By default, anything that changes data asks you first. Turn it off with --confirm for scripts.
  • A local audit log. Every tool call goes into a SQLite file. Replay with appctl history.

Will my app work with appctl?

The table shows what each source needs to be useful, and what to try if the first attempt does not give you the tools you wanted.

Source What it reads If the tools look wrong
--openapi A URL or file with an OpenAPI 2 or 3 spec Fix the spec; this is the most reliable source.
--django Your project folder. Needs Django REST Framework in INSTALLED_APPS. Make sure --base-url includes the API mount, for example http://host:8001/api.
--rails config/routes.rb for resources and db/schema.rb for fields. Only resources :foo inside a namespace block is parsed. If you publish a Swagger doc via rswag, use --openapi for richer types.
--laravel routes/api.php and files under database/migrations/. Only Route::apiResource entries are parsed. Manually defined routes are not.
--aspnet A swagger.json anywhere in the project (best), or a scan of [ApiController] files. Export swagger.json with curl .../swagger/v1/swagger.json -o swagger.json and re run sync for cleaner tool names.
--strapi src/api/*/content-types/*/schema.json files. Nested components are not expanded. Add them manually if you need them.
--supabase A URL that serves the PostgREST OpenAPI at /rest/v1. Row level security is not visible at sync time. Tools may fail at runtime if RLS blocks the anon key.
--db A Postgres or MySQL connection string. The agent can write SQL. Use a read only database user, or run with --read-only.
--url A login URL plus credentials. Only useful for session based HTML apps. If the site is a single page JavaScript app, this source cannot see the routes. Use its API directly.
--mcp An MCP server URL. Registers one passthrough tool. See the MCP page for details.

FAQ

Who is this for?

Two groups. First: developers who have an app in production and want to let their teammates ask questions of it without learning the API. Second: solo builders who want to chat with their own side project instead of clicking through it.

Do I need to learn any commands?

Only the person doing the one time setup runs appctl sync and appctl serve. After that, the chat window is plain English. Your teammates never see the CLI.

Does it work against my production app?

Yes. Give it a production OpenAPI URL, a production database connection string, or a live website URL. It uses the same API key or login you would use yourself, so it cannot do anything you could not do.

How do I pick an LLM?

For private data, run a local model with Ollama or vLLM. For quality, use OpenAI, Anthropic, or NVIDIA NIM. You change one line in .appctl/config.toml.

Where are my API keys stored?

In your operating system's keychain. On macOS that is Keychain. On Windows it is Credential Manager. On Linux it is the Secret Service daemon. Environment variables also work, for servers without a keychain.

My app is not in the sources list. What do I do?

If it has an OpenAPI spec, use --openapi. If it has a Postgres or MySQL database, use --db. If none of the above fit, you can write a plugin in Rust against appctl-plugin-sdk. The appctl-airtable plugin in the repo is around 100 lines of code.

What is the HTTP API?

appctl serve exposes a small set of endpoints. POST /run takes JSON like {"message": "..."} and returns the answer plus a list of tool events. WS /chat streams the same events as they happen. The deploy page has the full list.
